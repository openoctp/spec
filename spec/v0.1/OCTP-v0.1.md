# Open Contribution Trust Protocol v0.1

> **OCTP does not detect AI-generated code.** It does not decide what is good or bad. It creates a shared vocabulary so contributors can declare what their work is and maintainers can make informed decisions. The trust is in the declaration, the signature, and the reputation that accumulates over time.

---

## 1. Introduction

### 1.1 What is OCTP?

The Open Contribution Trust Protocol (OCTP) is a lightweight specification for cryptographically verifiable provenance declarations in open source software contributions.

In the AI era, open source maintainers face an unprecedented challenge: the volume of contributions has exploded while the cost of review remains constant. Every pull request could be human-written, AI-assisted, or fully AI-generated. OCTP solves this by creating a **standard vocabulary for trust** — a machine-readable envelope that travels with every contribution, declaring:

- **Provenance**: How was this code created? (human, AI-assisted, AI-generated)
- **Verification**: What automated checks has it passed?
- **Identity**: Who is asserting these claims?
- **Integrity**: Can we verify nothing was tampered with?

### 1.2 Why Now?

The "AI Slopageddon" is real. Maintainership burnout from AI-generated PR floods is forcing projects to close their doors:

- curl paused their bug bounty program due to AI-generated false reports
- Ghostty implemented "vouched users only" requirements
- tldraw auto-closes all external PRs by default

The problem isn't AI. The problem is **information asymmetry**: maintainers have no systematic way to know what they're reviewing.

### 1.3 Who is OCTP For?

**Contributors** who want to signal quality transparently:
- Declare honest provenance without stigma
- Show the care they've put into review
- Build reputation over time through signed contributions

**Maintainers** who need to triage efficiently:
- See at a glance: human/AI, review level, test status
- Filter queues based on declared characteristics
- Verify nothing was tampered with cryptographically

**Enterprises** who consume open source:
- Assess supply chain risk systematically
- Know which dependencies have AI-generated code
- Audit contribution provenance at scale

**AI Tooling** providers who want to be transparent:
- Declare when code was generated by their models
- Enable downstream provenance tracking
- Build trust through disclosure rather than hiding

### 1.4 Design Principles

1. **Declaration, not detection**: OCTP doesn't try to identify AI code. It lets contributors declare honestly what they made and how.

2. **Cryptographic binding**: Every declaration is signed. The signature binds the envelope to a specific git commit.

3. **Community ownership**: The spec belongs to the open source community, not any single company or individual.

4. **Layered verification**: Basic fields are required (provenance, identity, integrity). Richer signals are optional.

5. **Interoperability**: Designed to work with existing standards (CycloneDX, in-toto, SCITT, Sigstore).

## 2. Core Concepts

### 2.1 The Trust Envelope

An **envelope** is a JSON document containing structured information about a contribution. It is:

- **Portable**: Travels with the contribution, survives forks and clones
- **Verifiable**: Cryptographic signatures prove authenticity
- **Readable**: Both humans and machines can interpret it
- **Extensible**: Optional context allows project-specific additions

### 2.2 Provenance vs Verification

**Provenance** answers: "Where did this come from?"
- Who contributed it?
- Was AI involved?
- How much human review?
- Time invested?

**Verification** answers: "What checked it?"
- Did tests pass?
- Did static analysis find issues?
- Are dependencies secure?
- Any secrets leaked?

Provenance is **declared** by the contributor. Verification is **measured** by tools.

### 2.3 The Identity Layer

Every envelope carries a **developer_id** — an identifier linking the contributor to a keypair:

- `github:username` — Most common, uses github.com/username.keys
- `email:address` — Email-based identity
- `did:method:id` — Decentralized identifiers (future)

The private key signs the envelope. The public key verifies it.

### 2.4 Reputation Through History

Trust accumulates over time. A contributor with:
- 50 signed contributions
- Consistent high review levels
- No reverted code
- Transparent AI disclosure

...has more reputation than a first-time contributor.

OCTP enables this pattern without building a central reputation database — the signed envelopes themselves become the audit trail.

## 3. The Trust Envelope

### 3.1 Complete Field Reference

#### Top-Level Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `octp_version` | string | Yes | Spec version. Always "0.1" |
| `contribution_id` | UUID v4 | Yes | Unique identifier for this envelope |
| `timestamp` | ISO 8601 | Yes | UTC time of envelope generation |
| `repository` | string | Yes | Target repo, format: "platform/org/repo" |
| `commit_hash` | hex64 | Yes | SHA-256 of the commit this describes |
| `provenance` | object | Yes | Provenance declaration (Section 3.2) |
| `verification` | object | Yes | Verification results (Section 3.3) |
| `integrity` | object | Yes | Cryptographic integrity (Section 3.4) |
| `optional_context` | object | No | Free-form additional context (Section 3.5) |

#### 3.2 Provenance Object

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `method` | enum | Yes | How created: `human_only`, `ai_assisted_human_reviewed`, `ai_generated_human_reviewed`, `ai_generated_unreviewed` |
| `ai_tools` | array | No | AI tools used (required if method includes AI) |
| `human_review_level` | enum | Yes | Review depth: `none`, `glance`, `moderate_review`, `substantial_modification`, `complete_rewrite` |
| `human_review_duration_minutes` | int | No | Time spent reviewing |
| `developer_id` | string | Yes | Contributor identity |

**AI Tool Structure** (items in ai_tools array):

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `model` | string | Yes | Model name, e.g. "claude-sonnet-4-6" |
| `vendor` | string | Yes | Provider, e.g. "anthropic", "openai" |
| `version` | string | Yes | Model version or date |
| `usage_type` | string | Yes | How used: "drafting", "refactoring", "review", "debugging", etc. |

#### 3.3 Verification Object

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `tests_passed` | bool | Yes | Whether test suite passed (null if not run) |
| `test_suite_hash` | hex32-64 | No | Hash of test files for integrity |
| `static_analysis` | enum | Yes | Result: `passed`, `failed`, `skipped` |
| `static_analysis_tool` | string | No | Tool name and version |
| `dependency_check` | enum | Yes | Result: `passed`, `failed`, `skipped` |
| `novel_dependencies_introduced` | bool | Yes | Whether new deps were added |

#### 3.4 Integrity Object

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `payload_hash` | hex64 | Yes | SHA-256 hash of envelope without integrity section |
| `developer_signature` | base64 | Yes | ES256 signature of payload_hash |
| `signature_algorithm` | string | Yes | Always "ES256" in v0.1 |
| `signed_at` | ISO 8601 | Yes | UTC time of signing |

#### 3.5 Optional Context Object

All fields optional:

| Field | Type | Description |
|-------|------|-------------|
| `issue_reference` | string | Related issue/PR, e.g. "#447" |
| `self_assessed_confidence` | enum | `low`, `medium`, `high` |
| `areas_of_uncertainty` | string | Honest uncertainty notes |
| `time_in_codebase_minutes` | int | Time understanding codebase before contributing |

## 4. Verification Model

### 4.1 Verification Runners

OCTP uses **runners** — pluggable modules that execute specific checks:

**Example Runners:**
- `pytest_runner`: Execute test suite
- `ruff_runner`: Python linting
- `mypy_runner`: Type checking
- `bandit_runner`: Security scanning
- `semgrep_runner`: Pattern-based analysis
- `pip-audit_runner`: Dependency vulnerability check
- `detect-secrets_runner`: Secret/credential detection

### 4.2 Runner Profiles

Profiles group runners for different use cases:

**fast** (3-8 seconds)
- ruff, bandit, detect-secrets
- No network calls, essential checks only

**ci** (30-60 seconds)
- pytest, ruff, bandit, pip-audit, detect-secrets
- Balanced coverage for CI pipelines

**security** (30-60 seconds)
- bandit, pip-audit, detect-secrets, semgrep
- Focus on security concerns

**full** (2-4 minutes)
- All available runners
- Comprehensive verification

### 4.3 Runner Interface

All runners implement:

```python
class CheckRunner:
    name: str              # Runner identifier
    is_available() -> bool # Whether tool is installed
    run(repo_root) -> CheckResult

class CheckResult:
    passed: bool           # Whether check passed
    tool_name: str         # Tool name and version
    suite_hash: str|None   # Hash for integrity (tests)
    detail: str            # Human-readable summary
```

## 5. Cryptographic Model

### 5.1 Key Generation

Contributors generate an ES256 keypair:

```bash
# Private key (never shared)
~/.octp/keys/private.pem

# Public key (published to key registry)
~/.octp/keys/public.pem
```

Curve: NIST P-256 (secp256r1)

### 5.2 Envelope Hashing

Before signing, the envelope (minus integrity section) is serialized to JSON with:
- Keys sorted alphabetically
- No extra whitespace
- UTF-8 encoding
- No trailing newline

Then SHA-256 hashed:

```python
serialized = json.dumps(envelope_without_integrity, 
                       sort_keys=True, 
                       ensure_ascii=True)
payload_hash = sha256(serialized.encode()).hexdigest()
```

### 5.3 Signing

The private key signs the payload_hash:

```python
signature = private_key.sign(
    payload_hash.encode(),
    ECDSA(SHA256())
)
```

Signature is base64-encoded for JSON transport.

### 5.4 Verification

To verify:

1. Extract `payload_hash` and `developer_signature` from envelope
2. Recompute payload_hash from envelope (excluding integrity section)
3. Verify recomputed hash matches stored hash
4. Look up developer's public key from key registry
5. Verify signature using public key

If all checks pass, the envelope is authentic and untampered.

### 5.5 Commit Binding

The `commit_hash` field cryptographically binds the envelope to a specific git commit. This prevents:

- **Replay attacks**: Can't attach envelope to different code
- **Version confusion**: Envelope clearly belongs to specific commit
- **History manipulation**: Changing history breaks envelope validity

## 6. Security Considerations

### 6.1 Threat Model

#### Threat 1: Dishonest Declaration

**Scenario**: Contributor declares `human_only` when code was AI-generated.

**Impact**: Maintainer makes wrong trust decision based on false information.

**Mitigation**:
- Signature creates accountability — declaration is traceable to identity
- Social consequence is primary deterrent (reputation damage if discovered)
- No technical prevention possible without surveillance
- Explicitly acknowledge this limitation

**Risk**: Medium. Social enforcement, not technical.

#### Threat 2: Envelope Replay

**Scenario**: Attacher takes valid envelope from good PR and attaches to malicious PR.

**Impact**: Malicious code appears to have passed verification.

**Mitigation**:
- `commit_hash` field binds envelope to specific commit
- Verifier checks that envelope commit_hash matches actual commit
- Replayed envelope fails verification

**Risk**: Low. Prevented by design.

#### Threat 3: Envelope Tampering

**Scenario**: Modify fields after signing — change `ai_generated_unreviewed` to `human_only`.

**Impact**: False provenance information.

**Mitigation**:
- `payload_hash` covers all envelope fields
- Signature covers the hash
- Any modification invalidates signature
- Verification fails

**Risk**: Low. Prevented by design.

#### Threat 4: Key Compromise

**Scenario**: Developer's private key stolen, used to sign malicious contributions.

**Impact**: Attacker impersonates legitimate contributor.

**Mitigation**:
- Key revocation in registry (v0.2)
- Monitoring for unusual contribution patterns
- Multi-factor key protection

**Risk**: Medium. v0.1 acknowledges this as open question.

#### Threat 5: Weak Declarations

**Scenario**: Contributor genuinely confused about how to categorize their process.

**Impact**: Inaccurate but not malicious provenance.

**Mitigation**:
- Clear enum definitions with examples
- Good documentation and tooling
- `areas_of_uncertainty` field for honest ambiguity

**Risk**: Medium. Education and tooling help.

### 6.2 Privacy Considerations

- **developer_id** reveals contributor identity (by design)
- **timestamp** reveals when contribution was made
- **ai_tools** reveals tooling choices (competitive information)

All fields are opt-in at declaration time. The contributor controls what they disclose.

### 6.3 Key Management

**Private key storage**:
- Stored locally at `~/.octp/keys/private.pem`
- Never transmitted over network
- File permissions: 0600 (owner read/write only)

**Backup and recovery**:
- Contributor responsible for key backup
- Loss of private key = inability to sign new contributions
- Old signatures remain valid

**Rotation**:
- Generate new keypair if compromise suspected
- Update key registry with new public key
- Document rotation in commit messages

## 7. Compatibility

### 7.1 Relationship to Other Standards

**OCTP vs Sigstore/cosign**

Sigstore signs artifacts — binaries, container images, releases. OCTP signs the contribution process.

They are **complementary**:
- Use cosign to sign a release
- Use OCTP to sign every contribution that went into it
- Sigstore = signature on finished product
- OCTP = provenance trail of how it was made

**OCTP vs in-toto**

in-toto verifies software supply chain layouts — that each step was performed by authorized parties.

They are **complementary**:
- in-toto is organization/pipeline-facing
- OCTP is contributor-facing
- in-toto for formal supply chain policies
- OCTP for any developer submitting any PR
- An in-toto layout could require OCTP envelopes as one step

**OCTP vs CycloneDX AIBOM**

CycloneDX AIBOM documents what AI components are in software. OCTP documents how AI was used in creating contributions.

Different scope but **compatible**:
- AIBOM = inventory of AI in final product
- OCTP = provenance of AI in development process
- OCTP's `ai_tools` field designed for AIBOM cross-reference

**OCTP vs SBOM generally**

SBOMs are inventories of what's in software. OCTP is provenance of how software was created.

Related problems:
- SBOM = "What components do we have?"
- OCTP = "How did this specific contribution come to be?"

### 7.2 SCITT and RATS Compatibility

OCTP envelopes are compatible with IETF standards:

- **SCITT** (Supply Chain Integrity, Transparency and Trust): OCTP provides the attestation layer
- **RATS** (Remote ATtestation procedureS): OCTP's verification model aligns with RATS evidence structure

Future versions will provide explicit SCITT/RATS binding examples.

### 7.3 Version Migration

**From v0.1 to v0.2**:
- v0.2 will be backward compatible with v0.1
- New optional fields can be added
- Required fields will not change semantics
- Migration guide will be provided

**Breaking changes**:
- Only in major versions (v1.0, v2.0)
- Minimum 6 month deprecation notice
- Migration tooling provided

## 8. Implementation Guide

### 8.1 Building OCTP in Other Languages

Reference implementations should provide:

1. **Envelope model**: Data structures matching JSON Schema
2. **Git integration**: Read repository state
3. **Key management**: Generate ES256 keypairs, sign, verify
4. **Verification runners**: Plugin system for running checks
5. **CLI tool**: `sign`, `verify`, `init` commands
6. **Config parsing**: Read `.octp.toml`

### 8.2 Minimum Viable Implementation

The smallest OCTP implementation needs:

```
MUST generate valid JSON matching envelope schema
MUST use UUID v4 for contribution_id
MUST use ISO 8601 UTC for all timestamps  
MUST use SHA-256 for payload_hash
MUST use ES256 for signing in v0.1
MUST include all required fields
MUST reject envelopes with invalid signatures
MUST bind envelope to commit via commit_hash
SHOULD run at least one verification check
MAY implement additional optional runners
MAY extend optional_context with custom fields
MUST NOT modify required field semantics
```

### 8.3 Language-Specific Notes

**Rust**: Use `p256` crate for ES256, `serde` for JSON
**Go**: Use `crypto/ecdsa` from standard library
**JavaScript/Node**: Use `crypto` module or `elliptic` library
**Java**: Use BouncyCastle or standard JCE

### 8.4 Testing Compliance

A conformant implementation should pass:

1. Schema validation against JSON Schema
2. Signature verification test vectors
3. Hash computation test vectors
4. Full round-trip (create → verify)

Test vectors are provided in the test suite.

## 9. Conformance Criteria

### 9.1 Required Features

An OCTP v0.1 conformant implementation MUST:

**Envelope Generation**:
- [ ] Generate valid JSON matching schema
- [ ] Include all required top-level fields
- [ ] Include all required provenance fields
- [ ] Include all required verification fields
- [ ] Include all required integrity fields
- [ ] Use UUID v4 for contribution_id
- [ ] Use ISO 8601 UTC for timestamps
- [ ] Support all provenance method enum values
- [ ] Support all review level enum values

**Cryptography**:
- [ ] Generate ES256 keypairs (P-256 curve)
- [ ] Compute SHA-256 payload_hash correctly
- [ ] Sign payload_hash with ES256
- [ ] Encode signature as base64
- [ ] Verify signatures using public key

**Binding**:
- [ ] Read commit_hash from git
- [ ] Bind envelope to specific commit
- [ ] Prevent replay (verify commit_hash matches)

**Verification**:
- [ ] Verify envelope signature
- [ ] Verify payload_hash integrity
- [ ] Reject tampered envelopes
- [ ] Reject invalid signatures

### 9.2 Recommended Features

An implementation SHOULD:

- [ ] Support multiple verification runners
- [ ] Implement runner profiles (fast, ci, security, full)
- [ ] Provide interactive provenance collection
- [ ] Support non-interactive/CI mode
- [ ] Read `.octp.toml` configuration
- [ ] Support key registry lookups (GitHub)
- [ ] Handle network timeouts gracefully
- [ ] Provide human-readable output formatting

### 9.3 Optional Features

An implementation MAY:

- [ ] Add custom optional_context fields
- [ ] Implement additional verification runners
- [ ] Support decentralized identifiers (DIDs)
- [ ] Provide key rotation mechanisms
- [ ] Implement reputation scoring
- [ ] Support additional signature algorithms (future)

## Appendix B: Glossary

**Envelope**: The complete JSON document containing provenance, verification, and integrity.

**Provenance**: Information about the origin and creation process of a contribution.

**Verification**: Results of automated checks (tests, static analysis, security scans).

**Integrity**: Cryptographic proof that the envelope hasn't been tampered with.

**Runner**: A pluggable module that executes a specific verification check.

**Profile**: A named collection of runners (fast, ci, security, full).

**Payload Hash**: SHA-256 hash of the envelope without the integrity section.

**Developer ID**: Identity string linking contributor to a public key.

**Key Registry**: Service (e.g., GitHub) that stores public keys for verification.
